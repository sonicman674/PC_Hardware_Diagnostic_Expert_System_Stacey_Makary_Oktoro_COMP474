# Validation and Verification - Reproducible Test Cases

This file documents how to verify the D1 baseline and the D2 uncertainty layers independently.

## 1. Standard Load Procedure

Run the following before each test:

1. `(clear)`
2. `(load "facts.txt")`
3. `(load "rules.txt")`
4. Optional for interactive testing: `(load "driver.clp")`
5. `(reset)`

Helpful inspection commands:

1. `(facts)`
2. `(agenda)`
3. `(watch rules)`
4. `(watch facts)`

## 2. D1 Regression Tests

These tests verify that the original diagnosis behavior still exists in D2.

### Test D1.1 - Single crisp PSU diagnosis

Commands:

1. `(assert (reported-symptom (symptom-id no-power-at-all) (observed yes) (user-cf 1.0)))`
2. `(run)`

Expected:

1. `init-diagnostic-session` fires.
2. `diagnose-crisp-from-symptom` creates a `diagnostic-conclusion` with:
   - `scenario-id psu-failure`
   - `theory crisp`
   - `label strong`
3. A verification suggestion is printed because the score is high.

### Test D1.2 - Crisp strengthening now works

Commands:

1. `(assert (reported-symptom (symptom-id no-power-at-all) (observed yes) (user-cf 1.0)))`
2. `(assert (reported-symptom (symptom-id post-beep-codes) (observed yes) (user-cf 1.0)))`
3. `(run)`

Expected:

1. A crisp `psu-failure` conclusion is first created from the baseline rule.
2. `strengthen-crisp-psu-diagnosis` replaces the weaker crisp conclusion with:
   - `score 0.97`
   - `label very-strong`
   - `rule-id crisp-psu-combo`
3. This verifies that the old D1 multi-symptom limitation was fixed.

### Test D1.3 - Crisp storage strengthening

Commands:

1. `(assert (reported-symptom (symptom-id storage-not-detected) (observed yes) (user-cf 1.0)))`
2. `(assert (reported-symptom (symptom-id boot-device-not-found) (observed yes) (user-cf 1.0)))`
3. `(run)`

Expected:

1. A crisp `storage-connection-failure` conclusion exists.
2. `strengthen-crisp-storage-diagnosis` upgrades it to `label very-strong`.

### Test D1.4 - No evidence helper

Commands:

1. `(run)`

Expected:

1. `no-evidence-reported` fires once.
2. The console prints examples for asserting both `reported-symptom` and `sensor-reading`.

## 5. Cross-Theory Tests

### Test XT.1 - Same scenario supported by multiple theories

Commands:

1. `(assert (reported-symptom (symptom-id system-freezes-under-load) (observed yes) (user-cf 0.90)))`
2. `(assert (sensor-reading (name cpu-temperature) (value 100)))`
3. `(run)`

Expected:

1. A crisp `cpu-overheating` conclusion exists.
2. A probabilistic `cpu-overheating` conclusion exists.
3. A fuzzy `cpu-overheating` conclusion exists.
4. All three are traceable by `theory` and `rule-id`.

### Test XT.2 - Multiple modules at once

Commands:

1. `(assert (reported-symptom (symptom-id no-power-at-all) (observed yes) (user-cf 0.95)))`
2. `(assert (reported-symptom (symptom-id screen-artifacts) (observed yes) (user-cf 0.85)))`
3. `(assert (reported-symptom (symptom-id blue-screen-crashes) (observed yes) (user-cf 0.80)))`
4. `(assert (sensor-reading (name gpu-temperature) (value 92)))`
5. `(run)`

Expected:

1. Diagnoses from power, display, and memory modules can coexist.
2. No cross-module interference occurs.

## 7. Negative and Edge Cases

### Test NEG.1 - Invalid symptom identifier

Commands:

1. `(assert (reported-symptom (symptom-id nonexistent-symptom) (observed yes) (user-cf 1.0)))`
2. `(run)`

Expected:

1. The session starts.
2. No diagnosis is produced because no indicator or uncertain rule matches the unknown symptom.

### Test NEG.2 - Observed = no

Commands:

1. `(assert (reported-symptom (symptom-id no-power-at-all) (observed no) (user-cf 1.0)))`
2. `(run)`

Expected:

1. No symptom-based crisp, probabilistic, or fuzzy diagnosis fires because all symptom rules require `observed yes`.

### Test NEG.3 - Skipped fuzzy readings

Driver expectation:

1. Enter `-1` for any quantitative reading in `run-diagnosis`.

Expected:

1. The driver does not assert a `sensor-reading` fact for that variable.
2. The fuzzy rules that depend on the missing reading do not fire.
